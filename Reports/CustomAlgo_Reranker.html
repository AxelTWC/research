📐 Scoring Formula

At query time, every retrieved child chunk receives a composite score combining three orthogonal signals:

$$ \text{Score}_{\text{final}} = w_{\text{sem}} \cdot S_{\text{semantic}} + w_{\text{meta}} \cdot S_{\text{metadata}} + w_{\text{lex}} \cdot S_{\text{lexical}} $$
Equation (1): Final composite score for a child chunk
$$ w_{\text{sem}} + w_{\text{meta}} + w_{\text{lex}} = 1.00 $$
Equation (2): Macro-weight normalisation constraint
0.70 Dense Semantic $w_{\text{sem}}$
0.15 Structural Metadata $w_{\text{meta}}$
0.15 Exact Lexical $w_{\text{lex}}$
How the "top‑1 score" value works: All child chunk scores are min‑max normalised to $[0,1]$ across the 40 candidates, then children are grouped into parent chunks (up to 4 children per parent). The parent score is: $$ \text{Score}_{\text{parent}} = \max(\text{children}) + 0.15 \cdot \text{avg}(\text{children}) + 0.03 \cdot \text{consensus\_bonus} $$ The consensus bonus adds $+0.03$ per matching child (up to 5), so final scores can exceed $1.0$. Only the relative ordering matters — the absolute magnitude is not calibrated across different queries.

Semantic (0.70): Cosine similarity in embedding space — the "intuition" signal.
Metadata (0.15): Three internal tunables boost section matches, table intent, and numeric density.
Lexical (0.15): Exact keyword/BM25 overlap — the "barcode scanner" signal.

$$ S_{\text{metadata}} = \alpha \cdot I_{\text{section}} + \beta \cdot I_{\text{table}} + \gamma \cdot I_{\text{numeric}} $$
Equation (3): Metadata sub-score where $\alpha = 0.08$, $\beta = 0.07$, $\gamma = 0.05$

Both rerankers share the same candidate pool (step 2) and parent-merging logic (step 4). Only the scoring function in step 3 differs.

Step 2 — Exhaustive cosine similarity: Every stored child chunk embedding is compared against the query embedding. Haystack's InMemoryEmbeddingRetriever performs a brute-force scan of all vectors and returns the top-k most similar. Production pipeline uses top_k = 20; this benchmark widens it to 40 to give the reranker more candidates to reorder. No approximate (ANN / HNSW) indexing is involved — recall is exact, not approximate. Benchmark was run on NVIDIA GB10 (Blackwell GPU) via --gpus all Docker flag. Cross-encoder steady-state: ~0.37s/query on GPU vs ~7s on CPU.

🔍 Findings & Recommendations

✅ Hybrid reranker is fast and effective for in-domain queries
At 0.015s per query, the weighted hybrid formula (0.70+0.15+0.15) matches the cross-encoder's recall (1.00) for documents well-represented in embedding space. No accuracy is lost by using the simpler method.
✅ Both rerankers achieve perfect Recall@5 and MRR of 1.00
Every query returns the correct file (PCC_AC_MD.txt) in the top 5 results, and both methods rank it at position 1. The hybrid formula and the cross-encoder are equally effective at file-level retrieval for this document.
⚠️ Cross-encoder adds 25× latency even on GB10 GPU — no accuracy gain
On a GB10 (Blackwell GPU), the cross-encoder reaches ~0.37s/query steady-state (vs 0.015s for hybrid). The first query is slower (~161s) due to CUDA graph compilation and model download. Despite GPU acceleration, the hybrid formula delivers identical Recall@5 (1.00), MRR (1.00), and nearly identical scores (±0.01–0.06 difference). There is no retrieval quality benefit from deploying the cross-encoder.
✅ Intra-document chunk ordering differs between rerankers
Top-5 overlap averages ~3.4/5, meaning the two methods disagree on which passage within the same document is most relevant. The hybrid formula relies more on semantic vector proximity, while the cross-encoder detects subtler query-passage relevance — useful signal for future weight tuning.
